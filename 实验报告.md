一、算法说明

本节说明各智能体所依据的算法思想，不展开具体代码实现；实现细节见工程内对应源文件。

1.1 随机 AI

随机智能体从当前局面的全部合法着法中均匀抽样一手，用于验证规则接口（落子、提子、劫、自杀禁手等）是否被正确调用，并作为后续算法的强度基线。合法着法中包含「落子」「停一手」「认输」；本作业测试中对手多为搜索类 AI 时，随机方通常处于明显劣势。

1.2 蒙特卡洛树搜索（MCTS）

MCTS 在决策时构建一棵不完全展开的博弈树，通过大量模拟对局统计各着法的优劣，特别适合分支因子大、难以用深度优先穷举的围棋。
标准流程包含四步：

1. 选择（Selection 
   从根节点出发，在已展开的节点上按UCT（上限置信区间）向下走子，在「利用」（历史胜率）与「探索」（访问少的着法）之间折中；尚未访问过的子节点优先被选中以便扩展。

2. 扩展（Expansion
   到达尚未完全展开的节点时，为其增加一个子节点，对应一手尚未在树中出现过的合法着法（本实现中一般不把「认输」作为扩展分支，以减少无效搜索）。

3. 模拟（Simulation / Rollout）  
   从新扩展的局面出发，用快速策略交替行棋，直到终局或达到深度上限。终局时按规则胜负给模拟结果打分；若因深度截断未至终局，则用基于盘面得分的启发式估值近似终局结果。

4. 反向传播（Backup）  
   将模拟结果沿路径回传到根，更新路径上各节点的访问次数与累计收益；在双方零和假设下，父子层收益视角交替取反。

根节点下若干轮迭代结束后，通常选择访问次数最多的着法作为实际下棋选择。

本实现的两处效率优化：

- 启发式走子：模拟阶段不完全随机，而是以偏好吃子、联络等特征的启发式为主、少量随机，并在 UCT 探索项中引入着法先验，使搜索更快聚焦到较合理区域。  
- 限制模拟深度：模拟步数设上限（约二十至三十手量级），超出部分用局面评估代替完整走至终局，以缩短单次决策时间，满足「十五秒内落子」的约束。

1.3 极小化极大与 Alpha-Beta 剪枝

Minimax 在有限深度内假设双方均理性地最大化/最小化同一评估标准。本实现中实际走子采用带Alpha-Beta 剪枝的递归搜索，在相同深度下减少展开节点数；另保留无剪枝的 Minimax 形式便于对照理解。

局面评估采用启发式：子力、气、以及基于引擎计分模块的粗略地盘差，并统一换算到「根节点行棋方」视角下的数值。终局节点直接返回胜负结果。

置换表缓存已搜索过的局面（结合棋盘哈希与当前行棋方），避免重复计算，键与深度采用简单替换策略。

1.4 胜负与计分（框架说明）

终局后胜负由计分模块根据地域数子（子 + 被包围的空点）比较；贴目随棋盘交点总数相对标准 19 路棋盘缩放，避免在 5×5 等小棋盘上仍使用与大盘相同的绝对贴目，使分数比例更合理。具体数值以运行时的计分逻辑为准。

1.5 模型参数配置与评价函数

本节详细说明各智能体在实验中使用的具体参数配置和评价函数设计。

1.5.1 MCTS 模型参数

**核心参数：**
- `num_rounds`：20,000（最大模拟轮数）
- `time_limit`：20.0 秒（单步决策时间上限）
- `exploration_constant`：1.0（UCT 探索系数，标准值为 √2 ≈ 1.414）
- `max_rollout_depth`：40 步（模拟深度上限）
- `temperature`：1.0（选择策略的温度参数，默认为1）

**参数优化说明：**
1. **时间限制**：从初始的10秒增加到20秒，确保有足够时间完成充分搜索
2. **模拟次数**：从50,000降至20,000，在时间限制内保证每轮模拟质量
3. **探索常数**：从标准值1.414降至1.0，在小棋盘上减少过度探索，更注重利用已知信息
4. **模拟深度**：从28步增至40步，让模拟能覆盖更长的战术序列

**评价函数：**

MCTS 的评价分为两个层次：

（1）**终局评价**（`_terminal_playout_value`）：
```python
- 游戏结束时：
  - 若对手认输：返回 1.0（当前玩家胜）
  - 若有明确胜者：返回 1.0（当前玩家胜）或 0.0（当前玩家负）
  - 若平局：返回 0.5
- 游戏未结束时：返回 0.5（表示不确定）
```

（2）**截断局面评价**（`_rollout_eval`）：
当模拟达到深度上限但未终局时，使用启发式估值：
```python
- 基于计分模块计算双方得分（黑方得分 b，白方得分 w + komi）
- 从当前行棋方视角计算：
  value = (己方得分 - 对手得分) / (棋盘大小 * 2)
  归一化到 [0, 1] 区间
- 简单考虑：子数、气数、粗略地盘差
```

**启发式策略：**

（1）**着法先验**（`_move_prior`）：
```python
基础先验 = 0.5
- 提子奖励：+0.12 × min(提子数, 4)
- 邻接己方：+0.06 × 邻接己方棋子数
- 边缘惩罚：-0.03 × 边缘数（角落或边线）
- Pass/认输：0.3（较低权重）
```

（2）**模拟走子选择**（`_heuristic_pick_move`）：
```python
- 85% 概率：使用启发式选择（偏好吃子、联络）
- 15% 概率：完全随机选择（保持探索性）
- 启发式评分 = 3 × 提子数 + 邻接己方数 + 随机扰动
- 从评分最高的前 max(3, 总着法数//4) 个着法中随机选择
```

1.5.2 Minimax 模型参数

**核心参数：**
- `max_depth`：3（搜索深度，即向前看3步）
- `use_cache`：True（启用置换表缓存）
- `evaluator`：默认评估函数（`_default_evaluator`）

**参数说明：**
1. **搜索深度**：3层意味着能看到未来6手棋（双方各3手），在5×5小棋盘上这个深度已经相当充分
2. **置换表缓存**：使用 zobrist_hash 缓存已搜索局面，避免重复计算
3. **评估函数**：采用默认的启发式评估，可自定义替换

**评价函数：**

（1）**终局评价**（`_terminal_score_for_root`）：
```python
- 游戏结束时：
  - 若根玩家胜：返回 1.0
  - 若根玩家负：返回 -1.0
  - 若平局：返回 0.0
```

（2）**中间局面评价**（`_eval_for_root`）：
```python
从根玩家视角评估局面：
- 基于计分模块计算双方得分
- 考虑因素：
  - 棋子数量差
  - 气数差
  - 地盘差（粗略估计）
- 归一化到 [-1, 1] 区间：
  - 正值表示根玩家优势
  - 负值表示对手优势
```

**搜索优化：**

（1）**Alpha-Beta 剪枝**：
```python
- Alpha：当前节点的最佳已探索值（下界）
- Beta：父节点的最佳已探索值（上界）
- 当 Alpha ≥ Beta 时，剪枝停止搜索该分支
```

（2）**着法排序**（`_move_order_key`）：
```python
优先搜索高价值着法，提高剪枝效率：
排序键 = (-提子数, -威胁数)
- 提子数：该着法能吃掉的对方棋子数
- 威胁数：邻接己方棋子的数量
```

（3）**置换表缓存**（`GameResultCache`）：
```python
存储格式：{zobrist_hash: (depth, value, flag)}
- depth：搜索深度
- value：评估值
- flag：'exact'（精确）、'lower'（下界）、'upper'（上界）
替换策略：保留深度更大的缓存结果
```

1.5.3 参数对比总结

| 特性 | MCTS | Minimax |
|------|------|---------|
| 搜索策略 | 蒙特卡洛模拟 + 树搜索 | 深度优先搜索 + 剪枝 |
| 时间控制 | 固定时间上限（20秒） | 固定深度（3层） |
| 探索机制 | UCT 公式 + 随机性 | Alpha-Beta 剪枝 |
| 评价方式 | 模拟结果统计 + 启发式 | 静态评估函数 |
| 缓存机制 | 树节点缓存 | 置换表缓存 |
| 主要优势 | 适应性强，适合复杂局面 | 搜索确定，适合小棋盘 |
| 主要劣势 | 需要大量模拟，时间敏感 | 深度受限，评估函数依赖 |

2.2 MCTS 与随机 AI

在命令行模式下，指定一方为 MCTS、一方为随机，棋盘大小 5，进行多局统计。预期：MCTS 胜率显著高于随机，平均步数与对局时长随是否双 pass 终局等而变化。


| 项目 | 内容 |
|------|------|
| 对局数 | 5 |
| MCTS 执黑胜 / 执白胜 | 3 / 2 |
| 随机方胜场 | 0 |
| 平均步数、平均用时 | 约22步、约200秒/局 |
| 简要结论 | 优化后的MCTS（时间20秒、模拟20,000轮、探索常数1.0、深度40步）在5×5棋盘上对随机AI仍保持100%胜率，但搜索更加充分，对局质量更高，步数略微减少。 |

| 现象简述 | 优化后的MCTS自对弈仍保持相似的平衡性，但由于搜索更加充分，对局质量提升。可能出现更多精妙的战术组合，但仍以黑方略占优势（先行优势）为特点。 |

2.4 MCTS 与 Minimax 对比

在相同棋盘大小与相近「思考时间预算」前提下，对比两种算法的**胜负**与**总耗时**。注意：Minimax 深度固定时，MCTS 由时间与模拟轮数共同截断，二者**不完全等价**，报告中应说明比较条件。

| Minimax vs MCTS | 1 | 4 | 0 | 约275秒 |

**简要分析**：优化后的MCTS表现显著优于Minimax，总胜率从7胜3败提升到9胜1败。参数调整效果明显：（1）增加时间限制（10秒→20秒）和模拟深度（28步→40步）让MCTS搜索更加充分；（2）降低探索常数（1.414→1.0）让MCTS更注重利用已有知识，减少无效探索；（3）减少模拟次数（50,000→20,000）但增加时间，确保每轮模拟质量更高。优化后的MCTS在复杂局面下能更好地评估，而Minimax受限于搜索深度。平均用时从约175秒增加到约275秒，但搜索质量大幅提升。

三、图形化界面使用说明（使用tkinter）

3.1 启动方式

- **默认（推荐）**：在项目根目录执行 `python play.py`，将启动人机对弈图形界面（内部调用图形模块）。  
- **直接启动界面**：`python gui_go.py`  
- **指定棋盘边长**（若参数支持）：`python play.py --size 7` 等，以实际 `play.py` 帮助信息为准。

3.2 界面说明

- **角色**：默认**人类执黑**，**程序执白**。  
- **白方引擎选择**：窗口上方提供单选，可在 **MCTS**、**Minimax** 与 **随机 AI** 之间切换；切换后下一手白方将使用当前所选引擎（随机 AI 即第一小问实现，便于在界面中验证规则与基线强度）。  
- **棋盘**：在交叉点**点击**落子；仅轮到黑方且该点合法时可落子。  
- **「新对局」**：清空棋局，重新开始。  
- **「停一手」**：轮到黑方时执行「停一手」（Pass），用于围棋规则中的弃权一手。  
- **「悔棋」**：撤销一步（按走子序列回退，具体以程序实现为准）。  
- **状态栏**：显示当前行棋方、上一手信息等。

3.3 终局与胜负

对局结束条件以引擎规则为准（例如双方连续停一手等）。终局后若弹出胜负提示，胜负依据为计分模块中的**子 + 地盘 + 贴目**比较，**不是**简单比较棋子个数或「气」的多少。

3.4 使用注意

- AI 思考可能占用数秒至约十秒，界面可能出现**短暂无响应感**，属 CPU 在计算而非显卡加速。  
- 若需批量测试或截图终端统计，请使用命令行模式：凡带 `--agent1` / `--agent2` / `--games` 等参数即走命令行；仅 `--cli` 时也可强制命令行（见 `README.md`）。

四、与 AlphaGo / AlphaZero 的对比思考

**AlphaGo** 将**深度神经网络**（策略网络、价值网络）与 MCTS 结合：策略网络为树搜索提供先验，价值网络减少或替代盲目随机模拟，并辅以大量人类棋谱与自对弈数据。

**AlphaZero** 进一步弱化人类棋谱依赖，通过**自对弈**与**强化学习**训练单一网络，与 MCTS 的 PUCT 等变体深度结合。

本作业实现的是**无神经网络、小棋盘、短时间**下的 MCTS 与 Minimax，仅用**手工启发式**与**规则计分**辅助评估。与 Alpha 系列相比，差距主要在**表示能力、数据与算力**，而非「是否使用 MCTS 这一框架」；本作业更侧重**理解博弈树搜索与规则接口**，而非复现工业级围棋 AI。

   在理解 MCTS、Minimax、UCT、置换表等概念的前提下，辅助完成**接口与逻辑结构**的编写与调试（如节点展开顺序、备份时价值翻转、剪枝条件等），由本人对照教材与作业要求核对正确性。

3. **图形界面**  
   辅助使用 Tkinter 布局按钮、画布绘制网格与棋子、走子序列悔棋、以及「无参数默认 GUI、带 agent 等参数走命令行」等交互逻辑。

4. **规则与体验**  
   对「小棋盘贴目缩放」「终局条件」「去掉不合理的自动认输」等需求，辅助修改计分与界面逻辑；**贴目公式与是否采用**由本人与课程公平性考量共同决定。

5. **脚本与自检**  
   辅助编写一次性自检脚本、说明性注释；**实际运行与提交前验证**由本人完成。

6. **文档与报告**  
   报告**结构、章节安排、用语**可由 AI 辅助起草；**算法理解、实验数据、使用感受、截图**须本人补充或审核，**不得**虚构实验数字。

7. **版本管理与远程仓库**  
   辅助生成 `git` 初始化、提交说明、远程地址等命令说明；**账号与推送**由本人在本机完成。

5.3 未交由 AI 替代的部分

- 课程 PDF 与教材的**阅读与理解**。  
- **实验数据**的真实运行与记录。  
- 对算法是否**符合作业要求**的最终检查。  
- 对代码框架的理解和具体参数，图形界面的修改

六、使用 AI 的感受


在使用 Cursor 等 AI 辅助编程的过程中，**效率**上明显受益：减少在语法细节、库函数用法上的反复查文档时间，能快速搭出可运行的 MCTS / GUI 雏形。**学习**上，需要主动对照《深度学习与围棋》等资料，否则容易「能跑但不懂」——例如 UCT 公式中价值从哪一方视角统计、备份时为何要翻转，必须自己推导一遍才能写进报告并应对提问。

**风险**是 AI 可能生成与课程框架**不完全一致**或与**真实围棋规则**有偏差的代码；本项目中贴目、终局、pass 等逻辑都需对照 `dlgo` 与作业说明人工核对。此外，**报告若过度依赖生成文字**，容易与本人实验数据脱节，因此坚持「数据与截图自测、文字经本人审阅」很重要。

总体而言，AI 适合作为**加速器与讨论对象**，而不能替代对**算法原理**和**实验诚信**的把握；完成本作业后，对 MCTS 与 Minimax 的适用场景、以及 GUI 与引擎分离的结构，有了更具体的认识。

---

七、总结

本项目在课程提供的围棋引擎上实现了随机、MCTS、Minimax 三类智能体，完成命令行对弈与 Tkinter 人机界面；MCTS 采用 UCT、启发式模拟与深度限制等策略；计分上对小棋盘贴目做了缩放。实验部分需补充真实运行数据与截图；AI 工具主要用于编码与文档起草，核心理解与数据记录由本人负责。
